# راهنمای نصب و استفاده از پنل Xray VLESS + gRPC + REALITY

این پنل یک راه‌حول کامل و آماده‌ی Production برای مدیریت سرور Xray-core با پروتکل VLESS روی gRPC و REALITY است. یک پروسه‌ی واحد Node.js هم فرانت‌اند React، هم API، هم Endpoint اشتراک و هم خود Xray را اجرا می‌کند. مناسب Deployment روی Railway با کمترین مصرف منابع.

## معرفی کوتاه

این پنل شامل موارد زیر است:

- **احراز هویت مدیر**: رمز عبور bcrypt، نشست JWT، محدودسازی نرخ درخواست ورود
- **داشبورد**: وضعیت پروسه‌ی Xray، نمودار ترافیک امروز و ۷ روز اخیر، تعداد کلاینت‌های فعال
- **مدیریت کلاینت‌ها (CRUD)**: تولید UUID، نام، محدودیت ترافیک GB، تاریخ انقضا، فعال/غیرفعال
- **لینک اشتراک**: `GET /sub/:token` که لینک `vless://` را به‌صورت base64 برمی‌گرداند همراه با هدر `Subscription-Userinfo`
- **صفحه‌ی تنظیمات REALITY**: ویرایش پورت، هاست عمومی، `dest`، `serverNames`، `shortIds`، gRPC `serviceName`، Fingerprint، سطح لاگ، و تولید کلید REALITY
- **حسابداری ترافیک**: هر ۵ ثانیه از Stats API ایکس‌ری پرس‌وجو می‌کند و دلتاها را در دیتابیس ذخیره می‌کند؛ کلاینت‌هایی که به محدودیت یا انقضا می‌رسند به‌صورت خودکار غیرفعال می‌شوند
- **نمایشگر لاگ**: ۲۰۰ خط آخر لاگ دسترسی یا خطای Xray
- **رابط کاربری فارسی RTL**: فونت Vazirmatn، تم تیره با گرادیان بنفش به فیروزه‌ای، Glassmorphism

## استقرار روی Railway — گام به گام

### گام ۱: ارسال کد به GitHub

۱. پوشه‌ی `xray-panel` را به یک Repository در GitHub ببرید:

```bash
cd xray-panel
git init
git add .
git commit -m "Initial commit: Xray VLESS+gRPC+REALITY panel"
git branch -M main
git remote add origin https://github.com/YOUR_USERNAME/xray-panel.git
git push -u origin main
```

> توجه: پوشه‌های `node_modules/`، `client/node_modules/` و `client/dist/` نباید Push شوند. در صورت نیاز فایل `.gitignore` اضافه کنید.

### گام ۲: ساخت سرویس در Railway

۱. وارد Railway شوید و روی **New Project** کلیک کنید.
۲. گزینه‌ی **Deploy from GitHub repo** را انتخاب کنید.
۳. Repository پروژه را انتخاب کنید.
۴. Railway به‌صورت خودکار `Dockerfile` را تشخیص می‌دهد و Build را شروع می‌کند.

در اولین Build:
- فرانت‌اند React با Vite کامپایل می‌شود
- وابستگی‌های Production بک‌اند نصب می‌شوند
- باینری رسمی Xray-core (~۲۵ مگابایت) از GitHub Releases دانلود می‌شود

### گام ۳: اتصال Volume به مسیر `/data`

همه‌ی داده‌هایی که باید پس از Redeploy باقی بمانند در `/data` ذخیره می‌شوند:

۱. در صفحه‌ی سرویس، به تب **Settings** بروید.
۲. روی **+ Add Volume** کلیک کنید.
۳. در فیلد **Mount path** مقدار `/data` را وارد کنید.
۴. روی **Add** کلیک کنید.

این کار باعث می‌شود فایل SQLite، `xray-config.json`، لاگ‌ها و کلید REALITY پس از هر Redeploy حفظ شوند.

### گام ۴: تنظیم متغیرهای محیطی

در تب **Variables** سرویس، حداقل متغیرهای زیر را اضافه کنید:

| متغیر | مقدار پیشنهادی |
|---|---|
| `JWT_SECRET` | یک رشته‌ی تصادفی حداقل ۳۲ کاراکتری (مثلاً `openssl rand -hex 32`) |
| `ADMIN_USER` | `admin` |
| `ADMIN_PASSWORD` | رمز عبور دلخواه شما |

اگر `JWT_SECRET` و `ADMIN_PASSWORD` را تنظیم نکنید، پنل در اولین اجرا مقادیر تصادفی تولید می‌کند و **یک‌بار** در لاگ Deployment چاپ می‌کند. حتماً آن‌ها را ذخیره کنید و سپس به‌عنوان متغیر محیطی در Railway وارد کنید تا پس از Redeploy تغییر نکنند.

### گام ۵: گرفتن دامنه‌ی پنل

۱. در تب **Settings** سرویس، بخش **Networking** را پیدا کنید.
۲. روی **Generate Domain** کلیک کنید.
۳. Railway یک آدرس `*.up.railway.app` به شما می‌دهد که به `$PORT` (پورت HTTP پنل) روت می‌شود.
۴. آدرس را در مرورگر باز کنید و با نام کاربری و رمز عبور وارد شوید.

### گام ۶: فعال‌سازی TCP Proxy برای Xray (مهم)

پنل از HTTP Routing معمول Railway استفاده می‌کند، اما ترافیک VLESS یک **پورت TCP جداگانه** لازم دارد که Railway با TCP Proxy آن را در دسترس قرار می‌دهد:

۱. در تب **Settings** سرویس، بخش **Networking** را پیدا کنید.
۲. روی **Add TCP Proxy** (یا **Generate TCP Service** — نام در UI Railway ممکن است متفاوت باشد) کلیک کنید.
۳. Railway یک جفت `host:port` عمومی به شما می‌دهد، مثلاً:
   ```
   your-service-production.up.railway.app:12345
   ```
   این پورت TCP به `$XRAY_PORT` سرویس شما (پیش‌فرض `8443`) فوروارد می‌شود.
۴. این **host** و **port** عمومی را کپی کنید — در گام بعدی لازم داریم.

### گام ۷: تنظیم REALITY KeyPair و پیکربندی نهایی

۱. وارد پنل شوید و به صفحه‌ی **تنظیمات** بروید.
۲. در بخش **اتصال**:
   - **هاست عمومی (TCP Proxy)**: هاستی که Railway در گام قبل داد
   - **پورت عمومی (TCP Proxy)**: پورتی که Railway داد
3. در بخش **REALITY**:
   - روی **تولید کلید جدید** کلیک کنید. پنل دستور `xray x25519` را اجرا می‌کند و کلید عمومی و خصوصی را در `/data` ذخیره می‌کند.
   - فیلد **مخفی‌سازی (dest)** را به یک سایت معتبر تغییر دهید (مثلاً `www.microsoft.com:443`، `www.cloudflare.com:443`). این سایت باید از TLS 1.3 و HTTP/2 پشتیبانی کند و ترافیک بالایی داشته باشد.
   - فیلد **serverNames** معمولاً همان دامنه‌ی dest بدون `:443` است.
4. روی **ذخیره و ری‌استارت Xray** کلیک کنید.

> توجه: هیچ‌گاه مقدار پیش‌فرض `dest` را رها نکنید. در غیر این صورت ممکن است REALITY به‌درستی کار نکند.

## افزودن اولین کلاینت

۱. به صفحه‌ی **کلاینت‌ها** بروید.
۲. روی **کلاینت جدید** کلیک کنید.
۳. فرم را پر کنید:
   - **نام / یادداشت**: مثلاً `گوشی من`
   - **محدودیت ترافیک (GB)**: مثلاً `50` (اختیاری — خالی = نامحدود)
   - **تاریخ انقضا**: مثلاً `2026-12-31` (اختیاری)
   - **فعال**: تیک بخورید
4. روی **ذخیره** کلیک کنید.
5. در لیست کلاینت‌ها، روی آیکون **QR** کلیک کنید.
6. یک پنجره باز می‌شود که شامل:
   - QR Code قابل اسکن با اپلیکیشن کلاینت
   - لینک `vless://` قابل کپی
   - لینک اشتراک (Subscription URL)

لینک `vless://` شامل تمام پارامترهای لازم است:
```
vless://UUID@HOST:PORT?security=reality&sni=...&fp=chrome&pbk=...&sid=...&type=grpc&serviceName=GunService&mode=gun#REMARK
```

### اپلیکیشن‌های کلاینت پیشنهادی

- **اندروید**: v2rayNG، NekoBox
- **iOS**: Streisand، Foxray، Shadowrocket
- **ویندوز**: v2rayN
- **مک**: Foxray

کافی است لینک `vless://` را در یکی از این اپلیکیشن‌ها Import کنید یا QR را اسکن کنید.

## توضیح کامل تنظیمات REALITY

### مفهوم REALITY

REALITY یک پروتکل امن‌سازی است که ترافیک VLESS را شبیه ترافیک واقعی TLS به یک سایت معتبر می‌کند. برخلاف TLS معمولی، نیازی به گواهی SSL ندارد و کلاینت با SNI یک سایت دیگر (مثلاً `www.microsoft.com`) به سرور شما متصل می‌شود، در حالی که سرور شما این ترافیک را می‌فهمد و پاسخ می‌دهد.

### فیلدهای صفحه‌ی تنظیمات

| فیلد | توضیح |
|---|---|
| **پورت داخلی Xray** | پورتی که Xray روی آن گوش می‌دهد (env: `XRAY_PORT`). باید با TCP Proxy Railway همخوان باشد. |
| **هاست عمومی** | هاستی که Railway برای TCP Proxy به شما می‌دهد. |
| **پورت عمومی** | پورتی که Railway برای TCP Proxy به شما می‌دهد. |
| **مخفی‌سازی (dest)** | سایت مخفی‌سازی هدف. باید TLS 1.3 و HTTP/2 پشتیبانی کند. پیشنهادی: `www.microsoft.com:443`، `www.cloudflare.com:443`، `www.amazon.com:443`. |
| **serverNames** | لیست SNI‌های مجاز با کاما جدا می‌شوند. معمولاً همان دامنه‌ی dest بدون `:443`. |
| **shortIds** | شناسه‌های کوتاه برای تطابق کلاینت. مقدار خالی = هر ID پذیرفته شود. یا رشته‌ی هگز ۸ کاراکتری (مثلاً `0123456789abcdef`). |
| **gRPC serviceName** | نام سرویس gRPC. پیش‌فرض `GunService`. کلاینت باید همین مقدار را داشته باشد. |
| **Fingerprint پیش‌فرض** | اثر انگشت uTLS که در لینک کلاینت قرار می‌گیرد. `chrome` پیشنهادی است. |
| **سطح لاگ Xray** | `warning` پیشنهادی است. `debug` فشار زیادی به CPU و دیسک می‌آورد. |

### تولید کلید REALITY

کلید REALITY یک جفت x25519 (Private + Public) است:

- **Private Key**: روی سرور نگه داشته می‌شود، در `realitySettings.privateKey` قرار می‌گیرد
- **Public Key**: در لینک کلاینت (`pbk=`) قرار می‌گیرد

با کلیک روی **تولید کلید جدید**، پنل دستور `xray x25519` را اجرا می‌کند و هر دو کلید را در `/data` ذخیره می‌کند. **کلید خصوصی هرگز در پاسخ API برگردانده نمی‌شود**.

> هشدار: اگر کلید REALITY را تغییر دهید، تمام کلاینت‌های موجود باید لینک جدید دریافت کنند. این عمل را فقط در صورت نشت کلید انجام دهید.

## اشتراک‌گذاری با لینک Subscription

هر کلاینت یک **Subscription URL** منحصر به فرد دارد:

```
https://your-app.up.railway.app/sub/TOKEN
```

این URL:
- لینک `vless://` را به‌صورت base64 برمی‌گرداند
- هدر `Subscription-Userinfo` را شامل می‌شود که اطلاعات ترافیک را دارد:
  ```
  upload=1234567; download=9876543; total=53687091200; expire=1735689599
  ```
- اپلیکیشن‌هایی که Subscription را پشتیبانی می‌کنند (مثل v2rayN، NekoBox، Shadowrocket) به‌صورت دوره‌ای این URL را Fetch می‌کنند و در صورت تغییر لینک، کلاینت را به‌روزرسانی می‌کنند.

### مزایا

- اگر UUID کلاینت را Regenerate کنید، Subscription URL خودکار به‌روز می‌شود (توکن هم عوض می‌شود)
- اگر تنظیمات REALITY را تغییر دهید، Subscription URL به‌طور خودکار لینک جدید را برمی‌گرداند
- کلاینت می‌تواند مصرف ترافیک باقی‌مانده را در UI خود ببیند

## مدیریت ترافیک و محدودیت

### نحوه‌ی کار

هر ۵ ثانیه (قابل تنظیم با متغیر `stats_interval_ms`):

1. پنل دستور `xray api statsquery --server=127.0.0.1:$XRAY_API_PORT -reset` را اجرا می‌کند
2. خروجی به دلتاهای `up`/`down` برای هر کلاینت تبدیل می‌شود
3. دلتاها به ستون‌های `traffic_used_up` و `traffic_used_down` هر کلاینت اضافه می‌شوند
4. همان دلتاها به ردیف امروز در جدول `traffic_daily` اضافه می‌شوند
5. اگر کلاینت به `traffic_limit_gb` برسد یا `expires_at` گذشته باشد، به‌صورت خودکار `enabled = 0` می‌شود و پنل یک ری‌استارت Debounced برای Xray انجام می‌دهد تا کلاینت از Inbound حذف شود

### پرچم `-reset`

این پرچم باعث می‌شود هر Poll فقط ترافیک از Poll قبلی تا الان را برگرداند. اگر Xray ری‌استارت شود، ترافیک دوبار شمارش نمی‌شود.

### مشاهده‌ی ترافیک

- در داشبورد: نمودار ترافیک ۷ روز اخیر (به MB)
- در صفحه‌ی کلاینت‌ها: نوار پیشرفت ترافیک برای کلاینت‌های دارای محدودیت
- در Subscription URL: هدر `Subscription-Userinfo` شامل مقادیر `upload`، `download`، `total`، `expire`

## مشاهده‌ی لاگ‌ها

۱. به صفحه‌ی **لاگ‌ها** بروید.
۲. بین دو نوع لاگ می‌توانید سوییچ کنید:
   - **خطا** (Error log)
   - **دسترسی** (Access log)
۳. تازه‌سازی خودکار هر ۵ ثانیه فعال است (قابل غیرفعال کردن).
۴. فقط ۲۰۰ خط آخر نمایش داده می‌شوند — فایل کامل هرگز در حافظه بارگذاری نمی‌شود.

### سطح لاگ

در صفحه‌ی تنظیمات می‌توانید سطح لاگ را تغییر دهید:

| سطح | توصیه |
|---|---|
| `debug` | فقط برای عیب‌یابی — فشار زیادی به CPU و دیسک می‌آورد |
| `info` | اطلاعات کلی |
| `warning` | پیشنهادی Production |
| `error` | فقط خطاها |
| `none` | غیرفعال |

## متغیرهای محیطی

همه‌ی متغیرها اختیاری هستند — پنل در صورت نبود، مقادیر تصادفی امن تولید می‌کند.

| متغیر | پیش‌فرض | توضیح |
|---|---|---|
| `PORT` | `3000` | پورت HTTP پنل. Railway به‌صورت خودکار تزریق می‌کند. |
| `XRAY_PORT` | `8443` | پورت داخلی Inbound VLESS. با TCP Proxy Railway باز می‌شود. |
| `XRAY_API_PORT` | `10085` | پورت داخلی Stats API. هرگز به اینترنت باز نشود. |
| `DATA_DIR` | `/data` | مسیر ذخیره SQLite، کانفیگ، لاگ و کلیدها. Volume Railway اینجا مانت شود. |
| `XRAY_BIN` | `/usr/local/bin/xray` | مسیر باینری Xray (در Dockerfile دانلود می‌شود). |
| `STATIC_DIR` | `/app/client/dist` | مسیر فرانت‌اند Build شده. |
| `ADMIN_USER` | `admin` | نام کاربری مدیر. |
| `ADMIN_PASSWORD` | *(تصادفی)* | رمز عبور مدیر. در اولین اجرا چاپ می‌شود. |
| `JWT_SECRET` | *(تصادفی)* | راز امضای JWT. حتماً در Railway تنظیم شود. |
| `NODE_ENV` | `production` | محیط Node. |

## به‌روزرسانی Xray

برای ارتقای Xray-core به نسخه‌ی جدید:

۱. به [صفحه‌ی Releases در GitHub](https://github.com/XTLS/Xray-core/releases) بروید و آخرین نسخه را پیدا کنید.
۲. در `Dockerfile`، خط زیر را پیدا کنید:
   ```dockerfile
   ARG XRAY_VERSION=v25.6.30
   ```
۳. آن را به نسخه‌ی جدید تغییر دهید.
۴. Commit و Push کنید. Railway به‌صورت خودکار Redeploy می‌کند.

داده‌های Volume (کلاینت‌ها، تنظیمات، کلیدها) حفظ می‌شوند — فقط باینری تغییر می‌کند.

## توسعه‌ی محلی

برای توسعه و تست روی سیستم خودتان:

```bash
# نصب وابستگی‌های بک‌اند
cd xray-panel
npm install

# نصب وابستگی‌های فرانت‌اند
cd client
npm install
cd ..

# راه‌اندازی باینری Xray محلی
# در macOS: brew install xray  →  /opt/homebrew/bin/xray
# در لینوکس: دانلود از GitHub Releases
export XRAY_BIN=/path/to/xray
export DATA_DIR=./.data
mkdir -p .data

# اجرای بک‌اند (با قابلیت Reload خودکار)
npm run dev:server   # در یک ترمینال

# اجرای فرانت‌اند (با Proxy به :3000)
npm run dev:client   # در ترمینال دیگر
```

سپس به `http://localhost:5173` بروید. Vite به‌صورت خودکار `/api` و `/sub` را به `http://localhost:3000` Proxy می‌کند.

اگر `XRAY_BIN` در دسترس نباشد، بک‌اند هنوز اجرا می‌شود — فقط Xray استارت نمی‌خورد و داشبورد «متوقف» نشان می‌دهد. بقیه‌ی پنل برای تست UI کار می‌کند.

## عیب‌یابی

### Xray استارت نمی‌خورد

- بررسی کنید `XRAY_BIN` به مسیر درست باینری اشاره می‌کند
- لاگ Railway را بررسی کنید — پیام `[xray] spawn error` یا `[xray] binary not found` باید مشخص باشد
- اگر کانفیگ نامعتبر باشد، Xray با کد غیر صفر خارج می‌شود. فایل `/data/xray-config.json` را بررسی کنید.

### کلاینت نمی‌تواند وصل شود

۱. **TCP Proxy فعال است؟** در Railway Dashboard بررسی کنید.
۲. **هاست و پورت عمومی درست در تنظیمات وارد شده؟**
۳. **کلید REALITY تولید شده؟** در صفحه‌ی تنظیمات، بخش «کلید عمومی REALITY» باید مقدار داشته باشد.
۴. **`dest` یک سایت واقعی است؟** اگر روی پیش‌فرض `www.microsoft.com:443` است، سعی کنید `www.cloudflare.com:443` یا `www.amazon.com:443` را امتحان کنید.
۵. **`serverNames` با `sni` در لینک کلاینت همخوان است؟**
۶. **`shortIds` با `sid` در لینک کلاینت همخوان است؟** (مقدار خالی در `shortIds` = هر sid پذیرفته می‌شود)
۷. لاگ خطای Xray را در پنل بررسی کنید.

### ترافیک شمارش نمی‌شود

- بررسی کنید `XRAY_API_PORT` با پورت در `xray-config.json` (بخش `api` Inbound) همخوان باشد
- در لاگ Railway باید پیام `[stats] polling every 5000ms` را ببینید
- اگر کلاینت اتصال دارد اما ترافیک صفر است، ممکن است `email` کلاینت با `remark` در دیتابیس همخوان نباشد — پنل از `remark` به‌عنوان `email` در کانفیگ Xray استفاده می‌کند

### پس از Redeploy، کلاینت‌ها از بین رفتند

- Volume به `/data` مانت شده؟ در Railway Dashboard بررسی کنید.
- اگر Volume نباشد، فایل SQLite در Container موقت است و پس از Redeploy از بین می‌رود.

### نشست مدیر به‌طور مداوم منقضی می‌شود

- متغیر `JWT_SECRET` در Railway تنظیم شده؟ اگر نه، در هر Redeploy یک راز جدید تولید می‌شود و همه‌ی نشست‌ها باطل می‌شوند.

## نکات امنیتی

۱. **هرگز** فایل `.env` با مقادیر واقعی را Commit نکنید.
۲. **همیشه** `JWT_SECRET` را به‌صورت متغیر محیطی در Railway تنظیم کنید.
۳. **هرگز** رمز عبور ضعیف انتخاب نکنید — حداقل ۱۲ کاراکتر با ترکیب حروف، عدد و نماد.
۴. کلید خصوصی REALITY هرگز در پاسخ API برگردانده نمی‌شود. اگر می‌خواهید آن را ببینید، فایل `/data/panel.db` (جدول `settings`) را مستقیماً بررسی کنید.
۵. توکن Subscription یک راز ۳۲ کاراکتری hex است. در صورت نشت، UUID کلاینت را Regenerate کنید تا توکن جدید تولید شود.
۶. Rate Limiting روی `/api/auth/login` فعال است: ۱۰ تلاش در ۵ دقیقه به ازای هر IP.
۷. `helmet` روی Express فعال است. CSP غیرفعال شده تا فرانت‌اند Vite Build شده به‌درستی کار کند (استاندارد برای SPA).
۸. همه‌ی مسیرهای `/api/*` به جز `/api/login` نیاز به JWT دارند.

## جمع‌بندی

این پنل یک راه‌حول Production-Ready و کم‌مصرف برای مدیریت Xray با REALITY است. با حجم کمتر از ۱۰۰ مگابایت تصویر Docker و مصرف RAM حدود ۸۰ تا ۱۲۰ مگابایت در حالت Idle، تقریباً تمام منابع Railway شما برای ترافیک واقعی Proxy در دسترس باقی می‌ماند.

اگر مشکلی پیدا کردید یا قابلیت جدیدی می‌خواهید، می‌توانید مستقیماً کد را در `server/` یا `client/src/` تغییر دهید و Redeploy کنید — ساختار پروژه ساده و قابل فهم است.
